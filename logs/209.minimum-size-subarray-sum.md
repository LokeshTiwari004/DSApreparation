# Problem: Minimum Size Subarray Sum (#209)

## User's Approach

Variable-size sliding window with two pointers (`lo`, `hi`):
1. Initialize `win_sum = nums[0]`, `win_len = n+1`
2. Expand window by moving `hi` right while `win_sum < target`
3. When `win_sum >= target`, update minimum length, then contract from left by moving `lo`
4. Continue until `hi` reaches end and window can't expand further
5. Return `0` if no valid subarray found (`win_len == n+1`), else return `win_len`

---

## Correctness

✅ **Correct** — handles all cases:
- Single element ≥ target → returns 1
- Entire array sum < target → returns 0
- Multiple valid windows → finds minimum
- Constraints guarantee `nums` non-empty (1 ≤ nums.length), so `nums[0]` access is safe

---

## Pattern Analysis

**Pattern:** Sliding Window (Variable Size) — **Correct Label**

**Why it applies:**
- All numbers are positive (1 ≤ nums[i] ≤ 10⁴ per problem)
- With positive numbers, expanding window monotonically increases sum; contracting monotonically decreases sum
- This guarantees the two-pointer approach finds the optimal window without missing any candidate

**Canonical pattern:** Variable-size sliding window / "minimum window" subarray problem

## Code Quality

**Strengths:**
- Correct logic with proper window expansion/contraction
- Minimal variables, no extra data structures
- Handles edge cases correctly

**Issues:**
1. **Non-standard loop structure** — `while True` with multiple `break` statements is harder to follow than the canonical `for hi in range(n)` + inner `while` shrink loop
2. **Initialization quirk** — `win_sum = nums[0]` works due to constraints but isn't self-documenting; canonical version starts with `win_sum = 0` and handles empty window naturally
3. **Single-element window logic** — The `if lo < hi` / `else: break` branch for `lo == hi` case works but adds complexity; standard approach handles it uniformly

**Canonical Implementation:**
```python
class Solution:
    def minSubArrayLen(self, target: int, nums: List[int]) -> int:
        n = len(nums)
        lo = 0
        win_sum = 0
        min_len = n + 1
        
        for hi in range(n):
            win_sum += nums[hi]
            while win_sum >= target:
                min_len = min(min_len, hi - lo + 1)
                win_sum -= nums[lo]
                lo += 1
        
        return 0 if min_len == n + 1 else min_len
```

---

## Key Takeaways

1. **Variable-size sliding window** applies when all numbers are positive — expanding increases sum, contracting decreases it
2. **Canonical template:** `for hi in range(n)` to expand, `while condition` to contract — cleaner than `while True` with breaks
3. **Initialize sum = 0** rather than `nums[0]` — handles empty window naturally, avoids special-case logic
4. **Positive integers are the key** — this pattern fails with negative numbers (would need prefix sums + binary search / monotonic deque)
5. **Time complexity O(n)** because each index is visited at most twice — `hi` moves forward only, `lo` moves forward only
