# 55. Jump Game

## Problem
- **Title:** Jump Game
- **Number:** #55
- **Difficulty:** Medium
- **Link:** https://leetcode.com/problems/jump-game/

## User's Approach
Greedy: track `max_reachable`. For each index, if index > max_reachable, return False. Otherwise update `max_reachable = max(max_reachable, idx + nums[idx])`.

## Correctness
Correct. The greedy approach works because reachability is monotonic — if you can reach position i, any position before i is also reachable.

## Pattern Analysis
**Pattern: Greedy with reachability tracking.**
The key insight: we don't need to know the path, only the furthest reachable index. If at any point the current index exceeds max_reachable, we're stuck.

```python
def canJump(self, nums):
    max_reach = 0
    for i in range(len(nums)):
        if i > max_reach:
            return False
        max_reach = max(max_reach, i + nums[i])
    return True
```
This is the canonical greedy solution — O(n) time, O(1) space.

## Complexity
- **Time:** O(n) — single pass
- **Space:** O(1) — constant extra space

## Code Quality
Clean and canonical. 6 lines of logic, no unnecessary variables. The `id` variable name shadows the built-in `id()` function — use `i` instead. No header notes. This is the optimal solution.

## Key Takeaways
- Jump Game = greedy reachability (not DP, not BFS)
- If you can reach i, you can reach anywhere < i → monotonic property
- `i > max_reachable` = stuck condition → return False
- `max_reachable >= last index` = success condition at any point
- This pattern extends to #45 Jump Game II (minimum jumps — needs BFS/DP)
