# 104. Maximum Depth of Binary Tree

## Problem
- **Title:** Maximum Depth of Binary Tree
- **Number:** #104
- **Difficulty:** Easy
- **Link:** https://leetcode.com/problems/maximum-depth-of-binary-tree/

## User's Approach
Recursive with a height accumulator parameter `h`. Base case: `if not root: return h`. Recursive: `max(depth(left, h+1), depth(right, h+1))`.

## Correctness
Correct. The `h` parameter tracks depth from the root. Base case returns accumulated height. Handles empty tree (returns 0).

## Pattern Analysis
**Correct pattern: Recursive Tree Traversal (height calculation).**
Standard approach avoids the accumulator parameter:
```python
def maxDepth(self, root):
    if not root:
        return 0
    return 1 + max(self.maxDepth(root.left), self.maxDepth(root.right))
```
The `1 + max(...)` pattern is more idiomatic. Both approaches compute the same thing — the accumulator version threads state downward, while the standard version builds the result upward.

## Complexity
- **Time:** O(n) — each node visited once
- **Space:** O(n) — recursion stack up to tree height; worst case O(n) for skewed tree
- User stated **O(1) space — wrong.** The recursion stack uses O(n) space. O(1) would mean no extra memory regardless of input size.

## Code Quality
Correct but non-standard due to the `h` parameter. The standard `1 + max(...)` is cleaner — fewer parameters, more immediately readable. 10 min, no hints. Notes could include BFS approach as a variant.

## Key Takeaways
- Tree recursion template: base case → compute → return
- The standard approach uses `1 + max(left, right)` — no accumulator needed
- Space complexity from recursion is O(h) where h = tree height; don't call it O(1)
- BFS (level-order) is an alternative approach — O(n) time, O(w) space where w = max width
