# 153. Find Minimum in Rotated Sorted Array

## Problem
- **Title:** Find Minimum in Rotated Sorted Array
- **Number:** #153
- **Difficulty:** Medium
- **Link:** https://leetcode.com/problems/find-minimum-in-rotated-sorted-array/

## User's Approach
`while True` loop with adjacent-guard. Compares `nums[lower]` with `nums[cursor]` and `nums[cursor]` with `nums[upper]` to determine which half contains the rotation inflection point. When cursor == lower (adjacent), returns the smaller of `nums[lower]` and `nums[upper]`. If neither rotated condition fires, returns `nums[lower]` (array is normally sorted — no rotation).

## Correctness
Correct. The logic covers all cases:
- **Standard rotation** (pivot in left half): `nums[lower] > nums[cursor]` → min is in `[lower, cursor]`
- **Standard rotation** (pivot in right half): `nums[cursor] > nums[upper]` → min is in `[cursor, upper]`
- **No rotation**: neither fires → `nums[lower]` is the global minimum
- **Adjacent case**: returns the smaller of two remaining elements

## Pattern Analysis
**Correct pattern: Binary Search on condition (rotation point).**
The key insight: in a rotated sorted array, the minimum is at the inflection point where the order breaks. You don't need to find the target — you need to find where `nums[i] > nums[i+1]`. The binary search works because the rotated half is always the one where `nums[lo] > nums[hi]`.

Standard template (cleaner, no adjacent-guard):

```python
def findMin(nums):
    lo, hi = 0, len(nums) - 1
    while lo < hi:
        mid = lo + (hi - lo) // 2
        if nums[mid] > nums[hi]:      # min is in right half
            lo = mid + 1
        else:                           # min is in left half (mid included)
            hi = mid
    return nums[lo]
```
The key: compare `nums[mid]` with `nums[hi]`, not `nums[lo]`. This avoids needing to handle the "sorted" case separately.

## Complexity
- **Time:** O(log n) — correctly noted by user
- **Space:** O(1) — correctly noted

## Code Quality
Clean reasoning, good notes, correct complexity. 20 min, no hints — strong performance. The only issue is the persistent adjacent-guard pattern from the non-standard binary search template.

## Key Takeaways
- Rotated array BS: compare `mid` with `hi` to decide which half contains the inflection point
- `if nums[mid] > nums[hi]` → min is to the right; `else` → min is at or to the left of `mid`
- Using `hi` (not `lo`) as the comparison anchor avoids the "is array sorted?" special case
- The `lo < hi` pattern with `hi = mid` (inclusive of mid) works here because mid could be the answer
- This approach generalizes to all rotated array problems
