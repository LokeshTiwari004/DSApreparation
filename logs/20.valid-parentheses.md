# 20. Valid Parentheses

## Problem
- **Title:** Valid Parentheses
- **Number:** #20
- **Difficulty:** Easy
- **Link:** https://leetcode.com/problems/valid-parentheses/

## User's Approach
Stack + HashMap. Push opening brackets onto stack. On a closing bracket, check stack non-empty and top matches expected opener. After loop, return `True` only if stack is empty.

## Correctness
Correct. Handles: empty string (stack empty → True), unmatched closing (stack empty on pop), mismatched closing (top doesn't match), incomplete closing (non-empty stack after loop). All edge cases covered.

## Pattern Analysis
**Correct pattern: Stack.**
The LIFO nature of brackets (last opened = first closed) maps directly to a stack. HashMap maps each closer to its corresponding opener for O(1) lookup. This is the canonical stack problem.

## Complexity
- **Time:** O(n) — single pass
- **Space:** O(n) — stack grows to n in worst case (all opening brackets)
- User stated **O(1) space — wrong.** A stack that grows with input is O(n), not O(1). O(1) means fixed memory regardless of input size.

## Code Quality
Clean overall. Two minor issues:
1. `from collections import deque` is unnecessary — a plain Python `list` works as a stack (`list.append` / `list.pop` are both O(1)). `deque` is needed when popping from the left, not the right.
2. `if stack: return False` can simplify to `return not stack` — more Pythonic.

## Key Takeaways
- Stack is the canonical structure for bracket/parenthesis matching
- LIFO = last opened bracket must be the first closed
- HashMap for O(1) closer→opener lookup
- A stack that grows with input is **O(n) space**, not O(1)
- `return not stack` is cleaner than `if stack: return False / return True`
- Plain `list` is sufficient for stack operations — no need for `deque`
