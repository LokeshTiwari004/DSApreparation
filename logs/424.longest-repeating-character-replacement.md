# 424. Longest Repeating Character Replacement

## Problem
- **Title:** Longest Repeating Character Replacement
- **Number:** #424
- **Difficulty:** Medium
- **Link:** https://leetcode.com/problems/longest-repeating-character-replacement/

## User's Approach
Per-character sliding window. For each unique character in the string, builds the longest window where at most `k` other characters are replaced to match it. Uses a quota-based start (consume `k` replacements) then slides with adjustment logic.

## Correctness
Correct. The per-character approach tries every possible majority character. The sliding window logic maintains the constraint of ≤ k replacements.

## Pattern Analysis
**Pattern: Sliding Window with character-specific target.**
Standard single-pass approach tracks `max_freq` of any character in the window:
```python
def characterReplacement(self, s, k):
    count = {}
    left = 0
    max_freq = 0
    result = 0
    for right in range(len(s)):
        count[s[right]] = count.get(s[right], 0) + 1
        max_freq = max(max_freq, count[s[right]])
        if (right - left + 1) - max_freq > k:
            count[s[left]] -= 1
            left += 1
        result = max(result, right - left + 1)
    return result
```
Key insight: the optimal window's majority character is the one with max frequency in that window. `window - max_freq` = number of replacements needed. If that exceeds k, shrink.

## Complexity
- **Time:** O(n × charset) = O(26n) = O(n) — iterates per unique char
- **Space:** O(charset) — frequency dict
- Standard single-pass is O(n) time

## Code Quality
Correct but non-standard. The per-character iteration (O(26n) vs O(n)) adds unnecessary outer loop complexity. The sliding window adjustment logic (3-branch if/elif/else) is harder to follow than the standard `window - max_freq > k` condition. Early exit checks are extraneous. 30 min, no hints.

## Key Takeaways
- The standard solution uses a single sliding window tracking max_freq
- `window - max_freq > k` is the condition to shrink — no per-char iteration needed
- max_freq can only increase or stay the same → it never decreases even when shrinking
- This is the hardest "classic" sliding window problem to internalize
- O(n × charset) is valid but the single-pass O(n) solution is the interview target
