# 39. Combination Sum

## Problem
- **Title:** Combination Sum
- **Number:** #39
- **Difficulty:** Medium
- **Link:** https://leetcode.com/problems/combination-sum/

## User's Approach
Backtracking that picks from the **end** of the list. At each step: either include `remaining[-1]` (and keep it available) or exclude it (remove via `remaining[:-1]`). Base cases: `target == 0` (found), `target < 0` (overshot), `not remaining` (no candidates left).

## Correctness
Correct for all inputs. The two-branch decision (include last element / exclude last element) systematically explores all combinations. Since candidates are non-negative and unlimited, the include-branch keeps the element available while the exclude-branch removes it permanently.

## Pattern Analysis
**Pattern: Backtracking (two-choice decision tree).**
Standard approach uses a `start` index instead of list slicing:
```python
def combinationSum(self, candidates, target):
    result = []
    def backtrack(start, path, target):
        if target == 0:
            result.append(path[:])
            return
        for i in range(start, len(candidates)):
            if candidates[i] > target:
                break  # only works if sorted
            path.append(candidates[i])
            backtrack(i, path, target - candidates[i])
            path.pop()
    candidates.sort()
    backtrack(0, [], target)
    return result
```
The standard approach: (1) avoids slicing by tracking `start`, (2) uses append/pop instead of creating new lists, (3) uses `break` optimization with sorted candidates.

## Complexity
- **Time:** O(n^(T/M)) where T = target, M = min(candidates). Roughly O(n^target) in worst case.
- **Space:** O(T/M × n) for recursion stack + list copies due to slicing (non-standard overhead)
- User's "O(n^target), O(n*target)" is approximately correct

## Code Quality
Correct but has two non-standard choices:
1. **Picks from the end** — standard picks from the start with an index
2. **`remaining[:-1]` slicing** creates a new list at every exclude step, O(n) per call
The `path + [last]` approach creates new lists instead of append/pop, which is acceptable but less memory-efficient. No hints, 30 min — slightly over the 20-min rule.

## Key Takeaways
- Combination Sum = unbounded knapsack with backtracking
- The decision is: take current element (stay at same index) or skip it (move to next)
- Standard approach uses `start` index, not list slicing
- Sort + `break` when candidate > target is a common optimization
- This is the template for all "combination" problems (#39, #40, #216, #77)
