# 155. Min Stack

## Problem
- **Title:** Min Stack
- **Number:** #155
- **Difficulty:** Medium (Easy on some platforms)
- **Link:** https://leetcode.com/problems/min-stack/

## User's Approach
Two stacks: `stack` stores all values, `minstack` stores **indices** of minimum elements. On push, compares `stack[minstack[-1]]` with new value — if new value is smaller, pushes its index. On pop, checks if the top min-index exceeds the current top index (i.e., the minimum element was popped) and cleans up `minstack`.

## Correctness
Correct — passes all tests. The index-based approach correctly tracks minima because:
- Indices always point to valid minimum positions in the main stack
- After a pop, if the minimum was at the popped position, its index is now stale (> current top index) and gets removed from minstack

## Pattern Analysis
**Correct pattern: Auxiliary Stack (min tracking).**
Standard approach: maintain a separate stack that stores the **actual minimum value** at each level, not indices.

```python
def push(self, val):
    self.stack.append(val)
    if not self.minstack or val <= self.minstack[-1]:
        self.minstack.append(val)

def pop(self):
    if self.stack.pop() == self.minstack[-1]:
        self.minstack.pop()

def getMin(self):
    return self.minstack[-1]
```
This is simpler, more readable, and equally O(1) per operation. The index approach is clever but harder to reason about in an interview setting.

## Complexity
- **Time:** O(1) per operation — all operations are amortized O(1)
- **Space:** O(n) — two stacks grow with n

## Key Takeaways
- "Design a stack with min in O(1)" → auxiliary stack storing running minimums
- The standard solution is simpler than the index variant — use value comparison, not index comparison
- Always document pattern and complexity in notes after solving
- Each operation must be O(1) — no scanning or sorting
- O(n) space is acceptable for this problem, but O(1) extra space is not possible
