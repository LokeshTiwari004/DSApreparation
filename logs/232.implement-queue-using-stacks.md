# 232. Implement Queue using Stacks

## Problem
- **Title:** Implement Queue using Stacks
- **Number:** #232
- **Difficulty:** Easy
- **Link:** https://leetcode.com/problems/implement-queue-using-stacks/

## User's Approach
Two stacks: `first` and `second`. On `push`, transfers all elements from `second` to `first`, then pushes new element to `first`. On `pop`/`peek`, transfers all elements from `first` to `second`, then operates on `second`. `empty` checks both stacks are empty.

## Correctness
Correct — maintains FIFO order correctly. However, **transfers on every operation** making each push/pop/peek O(n) instead of amortized O(1).

## Pattern Analysis
**Correct pattern: Two Stacks (Queue Simulation).**
The standard approach uses amortized O(1) by only transferring when the output stack is empty:

```python
def __init__(self):
    self.input = []
    self.output = []

def push(self, x):
    self.input.append(x)

def pop(self):
    self.peek()
    return self.output.pop()

def peek(self):
    if not self.output:
        while self.input:
            self.output.append(self.input.pop())
    return self.output[-1]

def empty(self):
    return not self.input and not self.output
```
**Key difference:** transfer happens only when `output` is empty during `peek`/`pop`, not on every operation. Each element is moved at most twice (once into input, once into output) → O(n) total for n operations.

## Complexity
- **Time:** O(n) per operation in user's impl; amortized O(1) per operation in standard
- **Space:** O(n)
- User noted O(n) time — correct for their impl, but the standard is amortized O(1)

## Code Quality
Correct but inefficient for sequences of mixed operations. 100% runtime is misleading — LeetCode test cases are small enough that the difference doesn't show. In an interview, the expected answer is the amortized O(1) variant; this would be flagged.

## Key Takeaways
- Queue from two stacks: input stack for pushes, output stack for pops/peeks
- Only transfer when output is empty — this is the critical insight that makes it amortized O(1)
- "Amortized O(1)" means O(n) worst-case (when output is empty and everything transfers) but O(1) on average across n operations
- Each element enters input once and moves to output at most once — 2 pushes, 2 pops total per element
- 100% runtime on LeetCode doesn't mean the solution is optimal — always evaluate complexity independently
