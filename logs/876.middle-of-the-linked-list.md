# 876. Middle of the Linked List

## Problem
- **Title:** Middle of the Linked List
- **Number:** #876
- **Difficulty:** Easy
- **Link:** https://leetcode.com/problems/middle-of-the-linked-list/

## User's Approach
One-pass with a `middle` pointer and a `flag` that toggles. `head` advances every iteration. `middle` advances only when `flag = 1`, then `flag` toggles to 0. On the next iteration, `middle` stays (flag=0), then flag toggles back to 1. Effectively: middle moves once for every two head moves.

## Correctness
Correct for non-empty lists. Returns the second middle node for even-length lists (correct per problem spec). Edge case: null head → dereferences null on `head = head->next` (crashes). Single node → loop runs once, head becomes null, middle (still head) returned.

## Pattern Analysis
**Correct pattern: Middle of Linked List (Tortoise & Hare).**
User's flag-based approach is functionally equivalent to fast/slow pointers but non-standard:

```cpp
ListNode* middleNode(ListNode* head) {
    ListNode* slow = head, *fast = head;
    while (fast && fast->next) {
        slow = slow->next;
        fast = fast->next->next;
    }
    return slow;
}
```
Fast moves 2x, slow moves 1x. When fast reaches end, slow is at middle. Clean, no flag needed, and safe against null.

## Complexity
- **Time:** O(n) — single pass
- **Space:** O(1) — two pointer variables
- User stated O(n) / O(1) — **correct**

## Code Quality
Correct logic but three issues:
1. **C++ instead of Python** — interview language inconsistency
2. **Flag-based reinvention** — standard fast/slow is 5 lines and instantly recognizable. The flag approach requires mental tracing to verify.
3. **Null-safety** — `head = head->next` on a null head would crash

## Key Takeaways
- Middle of LL = fast/slow pointers: fast 2×, slow 1×. Hardcode this pattern.
- The fast/slow pattern also appears in: cycle detection (#141), palindrome check (#234)
- For even-length lists, the problem asks for the 2nd middle — slow ends up there naturally with the fast/slow pattern
- Always write in Python unless you're explicitly practising C++
- If an approach needs a flag variable, there's usually a cleaner pointer-based solution
