## User's Approach

Binary search on answer space `[max(nums), sum(nums)]`:
1. `lo = max(nums)`: minimum possible largest sum
2. `hi = sum(nums)`: maximum possible largest sum (one subarray)
3. Feasibility function `greedy(limit)`: check if array can be split into at most `k` subarrays where each subarray sum ≤ `limit`
4. If feasible → try smaller limit (`hi = mid`); else → need larger limit (`lo = mid + 1`)
5. Return `lo` (or `hi`)

Feasibility: iterate through nums, accumulate in current subarray; if adding the next number would exceed `limit`, start a new subarray and increment count. Return `True` if count ≤ `k`.

---

## Correctness

✅ **Correct** — handles all cases:
- `k = 1` → returns `sum(nums)`
- `k = len(nums)` → returns `max(nums)`
- All zeros → returns `0`
- Large values → `lo = max(nums)` ensures feasibility lower bound

Edge case: `k > len(nums)` — per constraints, `1 ≤ k ≤ nums.length`, so not possible.

## Pattern Analysis

**Pattern:** Binary Search on Answer — **Correct Label**

**Why it applies:**
- Monotonic: if limit `L` is feasible (can split with sum ≤ L), any `L' > L` is also feasible
- Need the minimum feasible limit
- Feasibility is decidable in O(n) with greedy checking

**Canonical pattern:** Minimize the maximum subarray sum → BS-on-answer with greedy feasibility

**Canonical Implementation:**
```python
class Solution:
    def splitArray(self, nums: List[int], k: int) -> int:
        def can_split(largest: int) -> bool:
            cur_sum = 0
            subarrays = 1
            for num in nums:
                if cur_sum + num > largest:
                    subarrays += 1
                    cur_sum = num
                else:
                    cur_sum += num
            return subarrays <= k

        lo, hi = max(nums), sum(nums)
        while lo < hi:
            mid = (lo + hi) // 2
            if can_split(mid):
                hi = mid
            else:
                lo = mid + 1
        return lo
```
