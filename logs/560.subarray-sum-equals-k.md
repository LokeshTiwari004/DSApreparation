## User's Approach

## Correctness

It works because: at position `i`, the prefix_sum map contains entries for positions `0` through `i-2` (it's populated AFTER the complement check). So single-element subarray `[i, i]` is not found via the complement path — the `num == k` check patches this. However this reasoning is fragile and non-obvious.

## Pattern Analysis
**Correct pattern: Prefix Sum + HashMap.**
Core insight: `sum(i, j) = prefix[j] - prefix[i-1]`. To find `sum(i, j) = k`, we need `prefix[i-1] = prefix[j] - k`. For each position `j`, check how many previous prefix sums equal `prefix[j] - k`.

The canonical implementation initializes `{0: 1}` before the loop (representing the empty prefix before any element), which eliminates the need for the special `num == k` case.

## Complexity
- **Time:** O(n) — one pass
- **Space:** O(n) — hashmap stores up to n distinct prefix sums
- User stated O(1) space — **wrong**. The `unordered_map` grows to O(n). Fix this.

## Canonical Solution
```python
from collections import defaultdict

def subarraySum(nums, k):
    prefix_sum = defaultdict(int)
    prefix_sum[0] = 1   # empty prefix: one way to have sum = 0
    prefix, answer = 0, 0
    for num in nums:
        prefix += num
        answer += prefix_sum[prefix - k]   # how many prior prefixes give sum = k
        prefix_sum[prefix] += 1
    return answer
```
The `{0: 1}` initialization handles: subarrays starting at index 0 (prefix before element 0 is 0) and single-element subarrays where `num == k` (prefix difference = current_prefix - 0-initialized-prefix = num).

- Space is O(n), not O(1) — the hashmap stores prefix sums
