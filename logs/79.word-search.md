# 79. Word Search

## Problem
- **Title:** Word Search
- **Number:** #79
- **Difficulty:** Medium
- **Link:** https://leetcode.com/problems/word-search/

## User's Approach
Backtracking DFS from every cell. Maintain a `seen` set of visited cells (flattened to 1D index). Explore 4 directions on match, backtrack on failure.

## Correctness
Correct. Visited tracking prevents reusing the same cell. Backtracking properly resets state. Bounds check and depth check are correct.

## Pattern Analysis
**Pattern: Graph DFS + Backtracking.** Standard grid DFS with backtracking. The search space is O(4^L) per starting cell, pruned by board boundaries and character mismatches.

## Complexity
- **Time:** O(m × n × 4^L) — each cell starts a DFS of up to 4^L paths
- **Space:** O(L + mn) — recursion stack depth L + seen set size up to mn

## Code Quality
Correct but a few issues:
- `id` shadows Python's built-in `id()` function — rename to `pos` or `key`
- Using a `set` of flattened indices works but the standard approach modifies the board in-place (e.g., `board[x][y] = "#"`) to save the O(mn) space
- Early bound check uses clean Python chained comparison `-1 < x < m` — nice
- No header notes (time/pattern/complexity)

Marked for re-attempt to practice the in-place board-marking pattern.

## Key Takeaways
- Grid DFS + backtracking is a common pattern
- In-place marking (`board[x][y] = "#" / temp`) avoids extra `seen` data structures
- Flattened index is fine but shadowing built-ins is a code smell
- Pruning order matters: check bounds + visited + char match before recursing
