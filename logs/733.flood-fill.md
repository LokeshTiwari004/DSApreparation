# 733. Flood Fill

## Problem
- **Title:** Flood Fill
- **Number:** #733
- **Difficulty:** Easy
- **Link:** https://leetcode.com/problems/flood-fill/

## User's Approach
DFS traversal on the grid starting from `(sr, sc)`. Uses a `closed` set for visited tracking. Bounds check → visited check → match base color → fill → 4-directional DFS. Early exit if base color equals target color.

## Correctness
Correct. The `base != color` early exit prevents redundant work. The `closed` set prevents infinite recursion on already-visited cells.

## Pattern Analysis
**Pattern: DFS on grid (graph traversal).**
Standard approach eliminates the `closed` set entirely:
```python
def floodFill(self, image, sr, sc, color):
    if image[sr][sc] == color:
        return image
    base = image[sr][sc]
    def dfs(r, c):
        if 0 <= r < len(image) and 0 <= c < len(image[0]) and image[r][c] == base:
            image[r][c] = color
            dfs(r-1, c)
            dfs(r+1, c)
            dfs(r, c-1)
            dfs(r, c+1)
    dfs(sr, sc)
    return image
```
Key insight: once a cell is changed to `color`, it no longer matches `base`, so subsequent DFS calls from neighboring cells naturally stop. No visited set needed.

## Complexity
- **Time:** O(M×N) — each cell visited at most once
- **Space:** O(M×N) — recursion stack (worst case for a long snake-like path)
- User's O(M×N) space is correct (recursion stack) but the `closed` set adds unnecessary extra O(M×N) space

## Code Quality
Correct but bloated by the `closed` set. The in-place modification is the entire point of flood fill — cells are self-visiting-marking. 20 min, no hints.

## Key Takeaways
- Flood fill = DFS/BFS on grid with in-place marking
- The in-place color change IS the visited tracking — no extra set needed
- Always check the base color != target color to skip unnecessary work
- Grid DFS template: bounds check → value check → recurse 4-dir
- BFS (queue) is also valid and avoids recursion depth issues
